## Implementation Approach

### 1. Type Extension (src/api-client.ts)

Extend `TurnSubmissionPayload`:
```typescript
export interface TurnSubmissionPayload {
  ship?: [number, number][];
  market?: [number, number][];
  relation_preference?: [number, number][];
  message?: [number, string][];  // NEW: [receiver_player_id, message_text]
}
```

### 2. UI Composition (src/game-ui.ts)

In `_buildTurnPanel`, add a "Messages" section after "Relation Preferences":

```html
<div class="turn-section">
  <div class="turn-section-title">Messages</div>
  <div class="turn-messages">
    <!-- Per-player text inputs -->
    <div class="turn-msg-row" data-player-id="${player.playerId}">
      <span class="turn-msg-name">${player.name}</span>
      <textarea class="turn-msg-input" data-player-id="${player.playerId}" 
                placeholder="Message to ${player.name}..." rows="2"></textarea>
    </div>
  </div>
</div>
```

Track message text in a module-level `Map<number, string>` (`_messageSelections`).

### 3. Submission Wiring (src/game-ui.ts)

In `_snapshotTurnSelections`, read all `.turn-msg-input` textareas and store
values in `_messageSelections`.

In `_handleTurnSubmit`, collect non-empty messages:
```typescript
const messages: NonNullable<TurnSubmissionPayload["message"]> = [];
section.querySelectorAll<HTMLTextAreaElement>(".turn-msg-input").forEach((textarea) => {
  const targetPlayerId = Number(textarea.dataset["playerId"]);
  const text = textarea.value.trim();
  if (Number.isFinite(targetPlayerId) && text.length > 0) {
    messages.push([targetPlayerId, text]);
  }
});
if (messages.length > 0) {
  payload.message = messages;
}
```

The server already accepts `message` keys in the submission dict — see
[`game_logic/core.py`](game_logic/core.py:143) lines 143-149. The key used is
`kw.message.TABLE_NAME`, which the server-side validation in
[`server/routes.py`](server/routes.py:971) already handles. The client must
send the payload key as `"message"` (matching the table name constant).
