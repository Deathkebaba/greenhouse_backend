## Plan

1. Strengthen diary image validation where P-1 stopped short of proof by adding end-to-end coverage for multiple images on a single entry plus closely related image error paths at the real api/web boundary.
2. Refactor diary tag filtering inside data_storage_service so filtering executes in SQL instead of after loading date-range entries into Rust memory, while preserving case-insensitive matching, original-case preservation, and `Any`/`All` semantics.
3. Validate the refined implementation with focused service-level checks for the SQL-backed filtering path and integration coverage that demonstrates the intended multi-image and image error behavior across service boundaries.

## Directive Compliance

- Directives checked: management/directives.md, management/architecture.md
- Compliance statement: The plan keeps api/web thin, preserves greenhouse_core as the shared contract layer, keeps diary filtering and persistence invariants owned by data_storage_service, and strengthens end-to-end validation where behavior crosses API, service, and database boundaries.
- Known risks or required exceptions: No directive exceptions requested. Main risks are preserving exact tag-filter behavior while changing query shape, keeping SQL readable under `Any` and `All` semantics, and avoiding brittle integration coverage while adding multi-image and error-path proof.

## Architecture Impact

- Existing architecture remains intact: greenhouse_core stays the shared DTO layer, data_storage_service remains the owner of diary persistence and filtering behavior, and api/web remains a forwarding gateway.
- No architecture.md update is required for this plan because the work tightens implementation and validation inside the existing diary ownership model rather than changing service boundaries or runtime topology.
