# Intent

Add diary media and tag capabilities without moving domain ownership out of the data storage service. Keep the web API as a forwarding layer, preserve shared DTO contracts in `greenhouse_core`, and verify the new behavior through end-to-end tests.

# Execution Outline

1. Extend diary contracts and persistence foundations.
   1. Add free-form tag support to diary request and response DTOs.
   2. Add image metadata and download references to diary responses.
   3. Add database schema support for diary images and tags.
2. Implement the owning data storage behavior.
   1. Persist images in the `data_storage_service` database.
   2. Support upload, download, and individual deletion of diary images.
   3. Support case-insensitive tag matching while preserving original casing for display.
   4. Extend date-range diary queries with tag filters and a mode parameter for `all` and `any` matching.
3. Extend the web API gateway.
   1. Preserve JSON diary create and update flows.
   2. Add separate diary image endpoints that forward to the owning service.
   3. Forward tag-aware diary queries through the existing diary surface.
4. Validate the full flow with integration coverage.

# Directive Compliance

- Directives checked: `management/directives.md`, `management/architecture.md`
- Compliance statement: The plan keeps persistence invariants and storage ownership in `data_storage_service`, updates shared contracts through `greenhouse_core`, and keeps `api/web` as a thin forwarding gateway.
- Known risks or required exceptions:
  - No approved directive exceptions are required.
  - Storing image bytes in PostgreSQL may increase database size and backup costs; the implementation should keep this explicit and test the behavior through the service boundary.
  - No upload size or MIME restrictions are planned in this scope, which is operationally permissive and may need a follow-up guardrail ticket if usage grows.
