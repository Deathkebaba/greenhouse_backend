## Plan

1. Extend shared diary contracts and persistence shape so images, image metadata, download routes, tags, and tag-filter query semantics are explicit and consistently owned.
2. Implement data-storage ownership for the new diary behavior, including schema migrations, binary image persistence, image metadata assembly, exact case-insensitive tag matching with original-case preservation, and image add/remove edit behavior.
3. Integrate the web API gateway so the public diary API forwards the new request and response contracts and emits backend-relative download URLs from the web surface.
4. Prove the feature end to end with integration coverage for create, edit, query, and download behavior across service boundaries.

## Directive Compliance

- Directives checked: management/directives.md, management/architecture.md
- Compliance statement: The plan preserves service ownership by keeping diary schema, binary storage, and filtering invariants in data_storage_service while using greenhouse_core as the single shared contract layer and keeping the web API thin.
- Known risks or required exceptions: No directive exceptions requested. Main implementation risks are migration shape for image/tag tables, binary payload size handling, and keeping query semantics identical across service and gateway layers.

## Architecture Impact

- Existing architecture remains intact: greenhouse_core owns DTOs, data_storage_service owns diary persistence and download behavior, api/web remains a forwarding gateway, and integration-tests remain the cross-service verification layer.
- No architecture.md update is required for this plan because the feature extends the current diary ownership model without changing service boundaries.
